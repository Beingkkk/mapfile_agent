2. 后端核心类关系
ConfigSession
会话根容器 | 单实例 | 内存驻留 | 无持久化
持有 / 引用

ConfigTree

  • _build_tree() — mappyfile dict → TreeNode/TreeLeaf
  • update_value() — 失焦更新 + 写回 params
  • add_object() / remove_object()
  • add_custom_property()
  • to_mappyfile_dict()

TreeNode / TreeLeaf

  • TreeNode: id, path, object_type, children[]
  • TreeLeaf: id, path, key, descriptor, value, user_modified, errors[]
  • 叶子对应 mappyfile 的一个 key-value
  • 对象层级: MAP → LAYER → CLASS → (STYLE | LABEL)

DialogueHistory

  • intent_message — 永久保留
  • messages — 当前关注点上下文（最多6轮）
  • set_focus() — 切换关注点时清空 messages
  • to_prompt_context()

TemplateMapper

  • 只读加载 mapguide_rules.json（运行时唯一规则源）
  • 合并来源：mapfile-schema-8-4.json + 8 个业务覆盖文件
  • get_field_descriptor() → FieldDescriptor
  • resolve_alias() — 别名转换
  • allows_custom_properties()
  • get_llm_context_summary()

FieldDescriptor

  • key, value_type, default, enum, min, max
  • phase (datasource/style/service/cache)
  • required, required_when, derived, custom, custom_desc

ValidationPipeline

  • Layer1: AliasResolver
  • Layer2: TypeChecker (Pydantic)
  • Layer3: SemanticValidator (dependencies.json)
  • Layer4: mappyfile.validate(version=8.4)
  • validate_field() — 失焦（1-3层）
  • validate_tree() — 完整（1-4层）

QAService

  • answer(session, question) → QAResult
  • 组装 L0-L5 Prompt → LLMClient.chat()
  • LLMOutput.parse() → UpdateResolver
  • apply updates → validate_tree()
  • 返回 qa_result

ExportService

  • export(session) → {filename: bytes}
  • mappyfile.dumps() → .map
  • MapCacheGenerator.render() → .xml（可选）
  • 前置校验：validation_state == "pass"

ImportService

  • import_mapfile(session_id, content, old_session) → (session, result)
  • mappyfile.loads() → dict
  • ConfigSession.from_mapfile_content() → 新建 session
  • validate_tree() → 四层完整校验
  • 解析失败 → 抛 MapfileParseError

PromptBuilder

  • _framework.j2 单模板
  • render(L0系统 + L1意图 + L2摘要 + L3关注点路径 + L4错误 + L5历史)
  • ~1500 tokens 总量控制

LLMClient

  • Anthropic SDK 封装
  • temperature=0.1
  • 指数退避 3× 重试
  • 总超时 ≤30s
  • 支持自定义 base_url

LLMOutput / UpdateResolver

  • parse(raw_json) → structured
  • 容错：非 JSON → 降级纯文本
  • resolve(update) → path（直接返回 update["path"]）
  • 路径不存在 → 静默忽略

MapCacheGenerator / Validator

  • Jinja2 模板渲染 mapcache.xml
  • MapCache 1.16.0 规则校验
  • 无需 MapCache 安装
  • 与 mappyfile 同层对等

4. WebSocket 数据流（八大交互场景）

☑️ 交互零：设置服务类型

1勾选 WMS/WFS/WCS / MapCache
2WS: set_service_types {services, mapcache_enabled}
3更新 session.service_types / mapcache_enabled
4ConfigTree._build_tree(service_types) 重建
5WS: tree_state {params, errors, service_types}
6树按新服务类型重新渲染

🔧 交互一：用户编辑参数（失焦）

1用户编辑 → 失焦
2WS: tree_update {updates: [{path, value}]}
3ConfigTree.update_value() 遍历应用
4ValidationPipeline.validate_field(service_types) (L1-L3)
5WS: tree_state {params, errors, can_export}
6更新树 + 错误标记 + 导出按钮状态

👆 交互二：用户切换关注点

1用户点击树节点/属性
2WS: focus_change {path}
3session.set_focus(path)
4历史清理（切换关注点时清空 messages）
5WS: focus_state {focus_param}
6右侧面板显示关注点

💬 交互三：用户提问

1用户输入问题 → 发送
2WS: question {text}
3QAService.answer(session, text)
4PromptBuilder.render(L0-L5) → LLMClient.chat()
5LLMOutput.parse() → UpdateResolver
6apply updates → validate_tree(service_types)
7WS: qa_result {bot_msg, updates, errors, can_export}
8追加对话 + 更新树 + 显示校验结果

✓ 交互四：手动校验全部

1点击「校验全部」
2WS: validate {}
3ValidationPipeline.validate_tree(service_types) (L1-L4)
4mappyfile.validate(version=8.4)
5WS: validation_result {state, errors, can_export}
6显示全局错误面板 + 更新导出按钮

📤 交互五：导出

1点击「导出」
2先执行 validate_tree（同交互四）
3通过后：ExportService.export(session)
4mappyfile.dumps() → .map + 可选 mapcache.xml
5WS: export_result {files: [{name, content_base64}]}
6弹出保存对话框

📂 交互六：导入 Mapfile

1点击「导入」→ Electron 文件选择
2主进程读取 → IPC → 渲染进程
3WS: import_mapfile { content }
4mappyfile.loads() → 失败则返回 error
5新建 session → _build_tree(service_types) → validate_tree(service_types)
6WS: import_result { success } + tree_state
7完全刷新树，清空历史，轮数归零

🔄 交互七：重置会话

1点击「重置」→ 确认
2WS: reset_session {}
3销毁旧 session，新建空 MAP
4WS: tree_state {空MAP, idle, [], {}, null, null}
5清空树 + 清空对话历史

6. 核心设计决策速查
决策点 结论 依据
前后端通信 统一 WebSocket（ws://localhost:18091/ws），禁止 HTTP/WS 混合；后端 → LLM 走 HTTP API 技术细节 §12
交互模式 用户主导配置树 + LLM 被动应答，无主动引导 需求输入 §1.2
初始界面 空 MAP 根节点（默认展开），无预置 LAYER；意图从 data/init_session_intent.json 读取，不强求用户输入 需求输入 §3.1 / 技术细节 §4.1
配置树渲染 分层渲染：ObjectCard（对象）+ FieldEditor（属性），不使用 n-tree；STYLE 内联 技术细节 §4.2 / §13.2
默认展开状态 MAP / LAYER / CLASS / STYLE 全部默认展开 技术细节 §4.2 TreeNode.expanded
寻址策略 LLM 参数更新使用扁平路径（如 layers.0.connectiontype），不使用行号；路径是稳定标识符 技术细节 §4.3
属性状态标记 `*` 必填 / `◆` 条件必填 / `D` 默认值 / `○` 可选 / `→` 推导 / `✎` 自定义；顶部显示图例说明栏 需求输入 §2.1
显示模式三档 三档：`all` 全部 / `required` 建议填（* 必填 + ◆ 条件必填）/ `strict` 仅必填（仅 *）。隐藏 `○` 可选和已填默认值的 `D` 字段（除非用户手动修改过）；有错误的始终显示 需求输入 §2.1
校验策略 4层：别名→类型→语义→mappyfile；失焦只跑1-3层，导出跑全部4层 需求输入 §4 / 技术细节 §9
删除确认 删除 LAYER / CLASS / STYLE 时弹出确认对话框 需求输入 §2.1
导出保存 后端通过 WS 返回 base64 文件内容；前端调用 Electron dialog.showSaveDialog 选择目录写入 需求输入 §5.3 / 技术细节 §7.6
MapCache 入口 作为 MAP 同级可选节点（树底部展开），非独立标签页 需求输入 §2.1
导入已有 Mapfile Electron dialog.showOpenDialog 选文件 → 主进程读取 → IPC → WS 发送内容 → 后端 mappyfile.loads() → 失败拒绝/成功重建 session（import_mode=True 抑制默认值回填）→ 自动四层校验 → 返回 tree_state 需求输入 §3.5 / 技术细节 §11.6
导入失败策略 分层策略：mappyfile 解析失败 → 完全拒绝；解析成功但校验失败 → 允许导入 + 标红修复 需求输入 §3.5
自定义属性识别 导入时 schema 未定义字段全部标记 custom=True，不维护白名单，由用户自行判断 技术细节 §11.6
字段搜索 顶部搜索框过滤当前树叶子，点击结果展开祖先节点、设置焦点、滚动到字段并触发高亮脉冲 Type-C
导入默认值抑制 import_mode=True 时缺失字段不回填默认值，保留原始字段的往返一致性；新节点 add_object() 仍预填默认值 Type-C
焦点切换 → 历史 切换关注点自动清空 QA messages（保留初始意图）；界面显示当前轮数，切换后归零；插入 divider 分隔线 技术细节 §5.2 / 需求输入 §2.2
LLM 提供商 MVP 只支持 Anthropic Claude；LLMClient 封装 SDK，不裸露 API 给上层 技术细节 §15.2
LLM 输出 JSON 解析失败降级为纯文本；路径不存在静默忽略；错误提示封装语气；发送问题时显示 loading 占位气泡 需求输入 §4.4 / 技术细节 §15.2
LLM 边界 不生成原始 Mapfile/XML；只输出自然语言 + params_update；所有规则后端处理 技术细节 §15.2
颜色格式 前端 ColorEditor 文本框直接显示 RGB 数组 `[0, 0, 255]`，直接存入 params；不使用 hex 技术细节 §15.1
Prompt 架构 单模板 _framework.j2；L0-L5 分级注入；~1500 tokens 总量 技术细节 §6 / §7.5
版本锁定 MapServer 8.4（mappyfile.validate 传 float）；MapCache 1.16.0；Python 3.11；Vue 3 + Naive UI 技术细节 §15.1
持久化 无持久化；ConfigSession 纯内存；重置=销毁重建 技术细节 §15.3

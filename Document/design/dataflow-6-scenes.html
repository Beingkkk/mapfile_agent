场景 0：设置服务类型
勾选 WMS/WFS/WCS / MapCache → 更新 session → 重建树 → 字段可见性同步
👤
Vue 前端
用户在顶部勾选/取消 WMS、WFS、WCS，或勾选 MapCache(WMTS/TMS)
📡
WebSocket → 后端
set_service_types { services: ["wms", "wfs"], mapcache_enabled: true }
⚙️
Python 后端
session.service_types = services
session.mapcache_enabled = mapcache_enabled
更新后触发 ConfigTree 重建（字段可见性重新计算）
🌲
Python 后端
ConfigTree._build_tree(params, service_types) — 按新的 service_types 过滤 METADATA/LAYER 字段可见性
├─ 取消勾选 WFS → gml_* / wfs_* 字段隐藏（值保留在 params 中）
├─ 取消勾选 WMS → wms_* 字段隐藏
└─ ows_* 始终可见
📐
Python 后端
ConfigTree.rebuild_line_map() — 行号重新分配
📡
WebSocket → 前端
tree_state { params_snapshot, validation_state, errors, line_map, service_types, mapcache_enabled }
🖥️
Vue 前端
配置树按新的服务类型重新渲染，隐藏/显示对应字段块
关键约束：隐藏字段的值保留在 params 中，只是不在 UI 渲染。取消勾选后重新勾选，之前填写的值会恢复显示。
🔧
场景 1：用户编辑参数（失焦触发）
字段失焦 → 实时校验（L1-L3）→ 行号重建 → 前端同步
👤
Vue 前端
用户在配置树中编辑某字段，失焦（blur）触发更新
📡
WebSocket → 后端
tree_update { updates: [{ path: "layers.0.name", value: "buildings" }] }
⚙️
Python 后端
ConfigTree.update_value(path, value) — 遍历 updates 逐个更新叶子节点并写回 params dict
🔍
Python 后端
ValidationPipeline.validate_field(path, service_types, full=false)
├─ L1 别名解析（如 "红色" → [255,0,0]）
├─ L2 类型校验（Pydantic）
└─ L3 语义校验（dependencies.json，含 service_types 条件）
📐
Python 后端
ConfigTree.rebuild_line_map() — 按 Mapfile 输出顺序重新分配行号
📡
WebSocket → 前端
tree_state { params_snapshot, validation_state, validation_errors, line_map, can_export }
🖥️
Vue 前端
更新配置树显示、错误标记（红框）、导出按钮可用状态
关键约束：失焦只执行 L1-L3 校验，不执行 mappyfile 语法校验（L4）——因为不完整 map 无法做语法校验。
👆
场景 2：用户切换关注点
点击树节点/属性 → 计算 focus_lines → 清空历史 → 同步前端
👤
Vue 前端
用户点击配置树中的属性行或对象节点，设为提问焦点
📡
WebSocket → 后端
focus_change { path: "layers.0.connectiontype" }
⚙️
Python 后端
session.set_focus(path)
├─ 计算 focus_lines（如 "L9-L23"）
├─ 若 focus_param 改变 → 清空 messages（保留 intent_message）
└─ 更新 session.focus_param / focus_lines
📡
WebSocket → 前端
focus_state { focus_param: "layers.0.connectiontype", focus_lines: "L9-L23" }
🖥️
Vue 前端
右侧面板显示关注点信息，QA 轮数归零
关键约束：切换关注点 = 开启新话题，之前关于 color 的讨论对 projection 问题没有帮助。intent_message 永久保留。
💬
场景 3：用户提问（LLM 问答）
用户输入问题 → Prompt 组装（L0-L5）→ LLM 调用 → 解析更新 → 校验 → 返回结果
👤
Vue 前端
用户在输入框输入问题，点击发送
📡
WebSocket → 后端
question { text: "connectiontype 是什么？" }
📝
Python 后端
history.add_message("user", text) — 记录用户问题
🧩
Python 后端
PromptBuilder.render(L0-L5)
├─ L0 系统角色（_framework.j2 ~200 tokens）
├─ L1 用户意图 + 当前 map 结构快照
├─ L2 关注点参数摘要
├─ L3 focus_param + focus_lines
├─ L4 当前校验错误（如有）
└─ L5 最近 4-6 轮对话
🤖
Python 后端 → LLM
LLMClient.chat(prompt) — Anthropic Claude, temperature=0.1, 3× 重试
🔧
Python 后端
LLMOutput.parse(raw_json) → UpdateResolver.resolve(line) → path
├─ 若返回 params_update → ConfigTree.update_value()
└─ 若 JSON 解析失败 → 降级为纯文本回答（不更新参数）
🔍
Python 后端
ValidationPipeline.validate_tree(service_types) — 四层完整校验 + rebuild_line_map()
📝
Python 后端
history.add_message("bot", answer) — 记录 LLM 回答
📡
WebSocket → 前端
qa_result { bot_message, params_update, validation_state, errors, can_export, line_map }
🖥️
Vue 前端
追加对话气泡、更新树（如有 LLM 修改）、显示校验结果、更新导出按钮
关键约束：LLM 输出不稳定时采用宽容降级——JSON 解析失败只展示纯文本；行号无法解析静默忽略该项。所有规则后端处理，LLM 不感知 schema。
✓
场景 4：手动校验全部
点击「校验全部」→ 遍历所有字段（L1-L3）→ mappyfile 语法校验（L4）→ 返回结果
👤
Vue 前端
用户点击底部 「✓ 校验全部」按钮
📡
WebSocket → 后端
validate {}
🔍
Python 后端
ValidationPipeline.validate_tree(service_types)
├─ 遍历所有叶子字段：L1 别名 + L2 类型 + L3 语义（含 service_types 条件）
└─ L4: mappyfile.validate(mf_dict, version=8.4)
📐
Python 后端
去重错误 → 计算 validation_state（pass / fail）→ rebuild_line_map()
📡
WebSocket → 前端
validation_result { validation_state, validation_errors, can_export }
🖥️
Vue 前端
显示全局错误面板，更新导出按钮状态（pass → 可用，fail → 禁用）
关键约束：手动校验和导出前校验共享同一套 validate_tree() 逻辑。L4 mappyfile 语法校验只在手动校验、导出前、导入后执行，字段失焦不执行。
📤
场景 5：导出配置
点击「导出」→ 前置校验 → 生成 .map + 可选 .xml → 返回 base64 → Electron 保存
👤
Vue 前端
用户点击底部 「📤 导出」按钮（仅校验通过时可用）
🔍
Python 后端（前置校验）
先执行 validate_tree()（同场景 4）——不可跳级导出
校验结果分支
失败：导出按钮保持禁用，返回错误列表，流程终止
通过：继续执行 ExportService.export()
📦
Python 后端
ExportService.export(session)
├─ mappyfile.dumps(tree.to_mappyfile_dict()) → .map（自动过滤 cache / _custom 内部节点）
└─ 若 session.mapcache_enabled == true → MapCacheGenerator.render() → .xml
📡
WebSocket → 前端
export_result { success: true, files: [{ name, content_base64 }] }
💾
Electron 主进程
前端调用 dialog.showSaveDialog → 用户选择保存目录 → 主进程将 base64 解码写入磁盘
🖥️
Vue 前端
显示「保存成功」提示，QA 面板追加系统消息
关键约束：导出条件链不可跳过：validation_state == "pass" ∧ 无 validation_errors。LLM 永远不生成原始 Mapfile/XML 文本，所有生成由后端确定性代码完成。
📂
场景 6：导入已有 Mapfile
点击「导入」→ 文件选择 → mappyfile 解析 → 分层处理 → 重建 session → 自动校验 → 同步前端
👤
Vue 前端
用户点击底部 「📂 导入 Mapfile」按钮
📂
Electron 主进程
dialog.showOpenDialog({ filters: [{ name: 'Mapfile', extensions: ['map'] }] })
📖
Electron 主进程
读取选中文件内容（UTF-8）→ 通过 IPC / preload 传给渲染进程
📡
WebSocket → 后端
import_mapfile { content: "MAP\nNAME \"city\"\n...\nEND" }
🔧
Python 后端
ConfigSession.from_mapfile_content(session_id, content, mapper)
├─ mappyfile.loads(content) → dict
├─ 失败 → 抛 MapfileParseError
└─ 成功 → 以解析结果新建 ConfigSession
解析结果分支（分层策略）
解析失败：WS 返回 import_result { success: false, error }，当前会话不受影响
解析成功：销毁旧 session，继续构建 ConfigTree → 自动四层校验
🌲
Python 后端
ConfigTree._build_tree(parsed_dict, service_types)
├─ Schema 已定义字段 → 正常 TreeLeaf（按 service_types 过滤可见性）
└─ 未知字段 → 统一标记 custom=True（显示 ✎ 图标）
🔍
Python 后端
ValidationPipeline.validate_tree(service_types) — 四层完整校验（L1-L4）
├─ 校验通过 → state="pass"，导出按钮可用
└─ 校验失败 → state="fail"，错误标红，用户可在应用内修复
📐
Python 后端
ConfigTree.rebuild_line_map() + 清空历史（保留 intent_message），focus_param 置空
📡
WebSocket → 前端
import_result { success: true }
→ 紧接着 tree_state { params_snapshot, validation_state, errors, line_map, can_export }
🖥️
Vue 前端
完全刷新配置树，显示自定义属性（✎）和校验错误（红框），清空问答历史，轮数归零
关键约束：导入 = 完全重置 + 从文件填充。文件读取必须由 Electron 主进程完成（安全沙箱限制）。未知字段不维护白名单，全部标记为 custom，由用户自行判断删除或保留。

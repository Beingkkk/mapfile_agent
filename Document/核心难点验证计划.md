# 核心难点验证计划

> 在正式动工前，针对三个技术风险最高的模块进行 Proof-of-Concept 验证。
> 验证通过后再进入全面开发，避免架构返工。

---

## 总览

| 验证项 | 优先级 | 依赖 | 预估时间 | 风险若失败 |
|--------|--------|------|----------|-----------|
| **V1: mappyfile validate 行为摸底** | P0 | 无 | 0.5 天 | 项目基础不成立 |
| **V2: LLM Prompt 输出稳定性** | P1 | 需 API key | 1~1.5 天 | 需大幅调整 LLM 交互设计 |
| **V3: ConfigTree 前端渲染** | P2 | 无 | 1 天 | 前端架构需调整 |

**建议执行顺序**：V1 与 V3 可并行 → V2 需等 V1 完成后（V1 结论影响 prompt 设计）。

**推荐安排**：
- Day 1：上午 V1 + 下午 V3（并行）
- Day 2：基于 V1 结论，跑 V2
- Day 3：V2 结果分析 + 三个验证的总结文档

## V1: mappyfile `validate()` 行为摸底

### 目标

摸清 `mappyfile.validate(mf, version=8.4)` 的**接受边界**：
- 哪些参数格式能过、哪些不能过
- `dumps()` 后的输出与输入的差异
- 形成一份**隐式要求清单**，指导 `to_mappyfile_dict()` 的实现

### 验证脚本

在 `SourceCode/spike/v1_mappyfile_validate.py` 创建：

```python
import mappyfile
import json

test_cases = [
    # 格式: (描述, 输入dict, 期望通过?)
    ("PROJECTION 数组", {"__type__": "map", "name": "test", "projection": ["init=epsg:3857"]}, True),
    ("PROJECTION 字符串", {"__type__": "map", "name": "test", "projection": "init=epsg:3857"}, False),
    ("颜色 RGB 数组", {"__type__": "map", "name": "test", "imagecolor": [255, 0, 0]}, True),
    ("颜色字符串", {"__type__": "map", "name": "test", "imagecolor": "255 0 0"}, None),
    ("LAYER 最小结构", {"__type__": "map", "name": "test", "layers": [{"__type__": "layer", "name": "l1", "type": "POLYGON"}]}, True),
    ("STYLE 缺必填字段", {"__type__": "map", "name": "test", "layers": [{"__type__": "layer", "name": "l1", "classes": [{"__type__": "class", "styles": [{"__type__": "style"}]}]}]}, None),
    ("METADATA 键值对", {"__type__": "map", "name": "test", "web": {"__type__": "web", "metadata": {"wms_title": "Test"}}}, True),
    ("Extent 数组", {"__type__": "map", "name": "test", "extent": [-180, -90, 180, 90]}, True),
    ("Extent 字符串", {"__type__": "map", "name": "test", "extent": "-180 -90 180 90"}, None),
    ("缺 name 字段", {"__type__": "map"}, False),
]

def test(description, mf, expected):
    try:
        errors = mappyfile.validate(mf, version=8.4)
        dumps = mappyfile.dumps(mf)
        passed = len(errors) == 0
        status = "✓" if passed else "✗"
        print(f"{status} [{description}] errors={len(errors)}")
        if errors:
            print(f"    errors: {errors[:2]}")
        print(f"    dumps[:120]: {dumps[:120]}...")
        return passed == expected if expected is not None else None
    except Exception as e:
        print(f"💥 [{description}] EXCEPTION: {e}")
        return False

for desc, mf, expected in test_cases:
    test(desc, mf, expected)
```

### 需覆盖的测试维度

| 维度 | 具体测试点 | 为什么重要 |
|------|-----------|-----------|
| **基本类型** | string、int、float、boolean 的接受格式 | 前端传的 JSON 值需要正确映射 |
| **颜色** | RGB 数组 `[R,G,B]` vs 字符串 `"R G B"` vs hex | 设计文档要求 RGB 数组，需验证 mappyfile 接受度 |
| **投影** | 数组 `["init=epsg:3857"]` vs 字符串 | 设计文档明确要求数组 |
| **Enum 值** | `connectiontype` 的合法值大小写 | POSTGIS vs postgis 是否都接受 |
| **嵌套对象** | LAYER/CLASS/STYLE/LABEL 的嵌套结构 | `to_mappyfile_dict()` 的递归逻辑 |
| **数组字段** | `layers`、`classes`、`styles` 是否必须是列表 | 单元素时能否省略列表 |
| **METADATA** | `metadata: {"key": "value"}` 的 key 限制 | 自定义属性是否会被 validate 拒绝 |
| **缺字段** | 哪些字段缺失会导致 validate 失败 | 区分 hard-required vs optional |
| **Extent** | 数组 `[minx, miny, maxx, maxy]` vs 字符串 | 常见 GIS 格式差异 |
| **自定义字段** | mappyfile 不认识但 MapServer 接受的字段 | `_custom` 扩展属性的可行性 |

### 成功标准

产出 `spike/v1_result.md`，包含：

1. **必须数组化的字段清单**（如 PROJECTION）
2. **enum 值严格大小写清单**
3. **validate 能检测的错误类型清单**
4. **validate 检测不到但 dumps 会出错的边界**
5. **`to_mappyfile_dict()` 的强制变换规则**

> 若发现 mappyfile 的宽容度低于预期（如大量字段要求严格格式），则 **ConfigTree 的序列化层需要做更多防御性转换**。

## V2: LLM Prompt 输出稳定性

### 目标

验证一个**简化版**的 PromptBuilder 能否让 Claude 稳定输出正确格式的结构化 JSON。

**不搭完整后端**，只写独立脚本测试 prompt → LLM → 解析链路。

### 验证脚本

在 `SourceCode/spike/v2_llm_prompt_stability.py` 创建：

```python
import json
import os
import anthropic

# 简化版 prompt 模板（只保留核心 L1-L3）
PROMPT_TEMPLATE = """你是一个 MapServer 配置助手。用户正在编辑一个 Mapfile。

## 当前配置
{context}

## 用户问题
{question}

## 回复格式
请严格用 JSON 回复，不要包含任何其他内容：
{{
  "action": "answer" | "update" | "question",
  "content": "回答内容（action=answer 时必填）",
  "params_update": [
    {{"path": "参数扁平路径，如 layers.0.name", "value": "新值", "reason": "修改原因"}}
  ]
}}

## 规则
- 参数路径使用点号分隔，数组用数字索引，如 layers.0.connectiontype
- 只修改你确定的参数，不确定的不要包含在 params_update 中
- 如果信息不足需要反问，action 用 "question"
- 值的类型要与参数类型匹配：字符串用引号、数字不用引号、布尔用 true/false、数组用 []
"""

def test_llm(question, context, trial=3):
    """同一个问题跑多次，统计稳定性"""
    client = anthropic.Anthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))
    prompt = PROMPT_TEMPLATE.format(context=context, question=question)

    results = []
    for i in range(trial):
        response = client.messages.create(
            model="claude-sonnet-4-6",  # 或 claude-opus-4-8
            max_tokens=1000,
            temperature=0.1,
            messages=[{"role": "user", "content": prompt}]
        )
        raw = response.content[0].text
        try:
            parsed = json.loads(raw)
            results.append({
                "trial": i + 1,
                "parse_ok": True,
                "action": parsed.get("action"),
                "paths": [u.get("path") for u in parsed.get("params_update", [])],
                "raw": raw[:200]
            })
        except json.JSONDecodeError:
            results.append({
                "trial": i + 1,
                "parse_ok": False,
                "raw": raw[:300]
            })
    return results

# 测试用例
if __name__ == "__main__":
    context = """MAP 名称: test
图层: LAYER[0] name=roads, type=POLYGON, connectiontype=POSTGIS, connection="host=localhost dbname=gis"
WEB: metadata={wms_title: "My Map"}"""

    questions = [
        "这个配置能发布 WMS 吗？",
        "把第一个图层的名称改成 highways",
        "我想加一个点图层",
        "WMS 的标题在哪里设？",
        "这个配置有什么问题？",
    ]

    for q in questions:
        print(f"\n=== 问题: {q} ===")
        results = test_llm(q, context, trial=3)
        for r in results:
            status = "✓" if r["parse_ok"] else "✗"
            print(f"  {status} Trial {r['trial']}: {r.get('action', 'N/A')} | paths={r.get('paths', [])}")
```

### 测试用例设计（10 个）

| # | 问题 | 期望 action | 考察点 |
|---|------|-------------|--------|
| 1 | "这个配置能发布 WMS 吗？" | answer | 纯回答，无 update |
| 2 | "把第一个图层的名称改成 roads" | update | 路径格式 `layers.0.name` |
| 3 | "我想用 PostGIS 数据源" | update/question | 多个关联字段（connectiontype + connection） |
| 4 | "颜色怎么设？" | answer | LLM 是否会教用户参数路径 |
| 5 | "帮我加一个叫 rivers 的图层" | update | 添加数组元素的路径表达 |
| 6 | "WMS 的标题在哪里设？" | update | METADATA 参数路径 `web.metadata.wms_title` |
| 7 | "这个配置有什么问题？" | answer | 基于当前上下文的分析能力 |
| 8 | "帮我创建一个 Mapfile"（空上下文） | question | 信息不足时的反问能力 |
| 9 | "把图层的投影改成 4326" | update | PROJECTION 参数的值格式（数组） |
| 10 | 连续提问，第 2 轮基于第 1 轮结果 | update | 上下文记忆和累积 update |

每个用例跑 **3 次**，统计：

| 指标 | 通过标准 | 说明 |
|------|----------|------|
| JSON 可解析率 | ≥ 90% | 30 次中 ≤ 3 次失败 |
| action 正确率 | ≥ 80% | answer/update/question 不混淆 |
| 路径格式正确率 | ≥ 80% | 点号+数字索引，如 `layers.0.name` |
| 值类型合理性 | ≥ 70% | 颜色给数组、布尔给 true/false 等 |

### 成功标准

| 结果 | 行动 |
|------|------|
| **可解析率 ≥ 90%** | 当前 prompt 设计可行，进入微调阶段 |
| **可解析率 70~90%** | 需增加 few-shot 示例或调整 prompt 结构 |
| **可解析率 < 70%** | prompt 框架需重新设计（如减少单次输出量、拆分多次调用） |

**额外产出**：记录 LLM 最容易出错的 path 格式，用于 UpdateResolver 的**宽容解析**（如自动转换 `layers[0]` → `layers.0`）。

## V3: ConfigTree 前端递归渲染

### 目标

验证**不用 `n-tree`** 的情况下，自定义递归组件能否流畅渲染一棵 50+ 字段的配置树，并支持内联编辑。

**不连后端**，用 Vite + Vue 3 + Naive UI 创建最小项目，用 `mapguide_rules.json` 生成静态数据。

### 项目结构

```
spike/v3-config-tree/
├── package.json              # vue@3 + naive-ui + vite
├── vite.config.js
├── index.html
└── src/
    ├── main.js
    ├── App.vue               # 根组件，加载静态数据
    ├── components/
    │   ├── ConfigTree.vue    # 根组件，管理展开状态
    │   ├── ObjectCard.vue    # 递归容器（MAP/LAYER/CLASS/STYLE/LABEL）
    │   └── FieldEditor.vue   # 叶子编辑器（按 value_type 映射控件）
    └── data/
        └── mock_tree.js      # 从 mapguide_rules.json 生成的静态树
```

### mock_tree.js 生成

从 `mapguide_rules.json` 的 `object_types.MAP` 提取字段定义，构造带模拟值的树：

```javascript

    { type: 'field', name: 'status', value: true, valueType: 'boolean', phase: 'datasource' },
    { type: 'field', name: 'extent', value: [-180, -90, 180, 90], valueType: 'array', phase: 'datasource' },
    { type: 'field', name: 'imagecolor', value: [255, 255, 255], valueType: 'color', phase: 'style' },

      type: 'object',
      objectType: 'LAYER',

      phase: 'datasource',
      children: [
        { type: 'field', name: 'name', value: 'roads', valueType: 'string', phase: 'datasource', required: true },
        { type: 'field', name: 'type', value: 'POLYGON', valueType: 'enum', allowed: ['POINT', 'LINE', 'POLYGON'], phase: 'datasource' },
        { type: 'field', name: 'connectiontype', value: 'POSTGIS', valueType: 'enum', allowed: ['POSTGIS', 'OGR', 'SHAPEFILE'], phase: 'datasource' },

    }
```

### 需要验证的交互

| 验证点 | 具体操作 | 成功标准 |
|--------|----------|----------|
| 递归渲染 | 展开 MAP → LAYER → CLASS → STYLE | 层级正确，无无限递归 |
| 字段数量 | 渲染整棵 MAP 树（约 50~80 个字段） | 无明显卡顿（首次渲染 < 200ms） |
| 内联编辑 | 点击文本字段 → 输入新值 → blur | 值更新、无页面跳动 |
| 控件映射 | enum → 下拉、boolean → 开关、int → 数字输入 | 控件正确渲染，值绑定正确 |
| 展开/折叠 | 点击箭头展开/折叠子节点 | 状态正确，动画流畅 |
| 焦点切换 | 点击不同字段，观察焦点状态 | 高亮正确，无焦点丢失 |
| 服务过滤（模拟） | 隐藏/显示部分字段 | 布局不崩，被隐藏的字段值保留 |
| 颜色编辑器 | RGB 数组输入 | 三个数字输入框绑定到数组 |

### 性能基准

使用 Chrome DevTools Performance 面板测量：

| 指标 | 目标值 |
|------|--------|
| 首次渲染（50 字段） | < 200ms |
| 展开/折叠响应 | < 50ms |
| 内联编辑输入延迟 | < 16ms（60fps） |
| 内存占用（稳定后） | < 额外 20MB |

### 成功标准

1. Chrome DevTools 测量全部通过
2. 递归组件代码 < 200 行，逻辑清晰
3. 50 个字段的树交互无明显卡顿
4. Naive UI 控件能覆盖所有 `value_type`（string/int/float/boolean/enum/array/color）

> 若性能不达标，考虑引入 `v-once`、虚拟滚动或延迟加载深层节点。

## Go / No-go 决策标准

| 条件 | 决策 |
|------|------|
| V1 通过 + V2 可解析率 ≥ 90% + V3 性能达标 | **Go**：按设计文档全面开发 |
| V1 通过 + V2 可解析率 70~90% + V3 性能达标 | **Go with caution**：先优化 prompt，再动工 |
| V1 失败（mappyfile 行为与设计假设严重冲突） | **No-go**：重新评估 ConfigTree 序列化策略 |
| V2 可解析率 < 70% | **No-go**：重新设计 LLM 交互架构（如拆分多次调用） |
| V3 性能不达标（> 500ms 渲染） | **No-go**：改用 `n-tree` 或引入虚拟滚动 |
